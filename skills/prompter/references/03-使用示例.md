# 使用示例（多示例提示）

## 为什么示例有效？

通过提供具体示例，可以让 Claude 更准确地理解期望的输出模式、风格和格式。这比单纯的文字描述更直观、更有效。

## 示例的力量

### 单一示例 vs 描述

**仅描述❌:**
```
用友好的语气回复客户投诉
```

**提供示例✅:**
```
用友好的语气回复客户投诉，参考以下示例：

示例：
客户: "我的订单已经延迟 3 天了！"

回复: "非常抱歉给您带来不便！我理解等待是令人沮丧的。让我立即查看您的订单状态，
并尽快为您解决这个问题。您的订单号是多少呢？"
```

## 多示例提示模式

### 模式 1: 输入-输出对

适用于转换、分类、格式化任务：

```xml
<示例>
输入: "Python is awesome!"
输出: {"sentiment": "positive", "confidence": 0.95}

输入: "This is terrible."
输出: {"sentiment": "negative", "confidence": 0.90}

输入: "The weather is okay."
输出: {"sentiment": "neutral", "confidence": 0.75}
</示例>

现在处理: "I love this product!"
```

### 模式 2: 模板展示

适用于创作、生成任务：

```xml
<示例>
产品: 无线耳机
标题: 🎧 沉浸音质，自由畅听 | XYZ 无线耳机限时特惠

产品: 智能手表
标题: ⌚ 健康管家，腕上科技 | ABC 智能手表新品上市
</示例>

请为以下产品创建标题：
产品: 蓝牙音箱
```

### 模式 3: 渐进式复杂度

从简单到复杂，展示不同情况：

```xml
<示例>
简单句: "我喜欢编程"
分析: 主语(我) + 谓语(喜欢) + 宾语(编程)

复杂句: "虽然很难，但我喜欢编程"
分析: 让步状语从句(虽然很难) + 主语(我) + 谓语(喜欢) + 宾语(编程)

复合句: "我喜欢编程，因为它很有趣，而且能解决问题"
分析: 主句(我喜欢编程) + 原因状语从句(因为它很有趣) + 并列从句(而且能解决问题)
</示例>
```

## Few-Shot 学习技巧

### 3-5 个示例通常最有效

**2 个示例 - 可能不够:**
```
示例 1: ...
示例 2: ...
```

**3-5 个示例 - 黄金数量:**
```
示例 1: ...
示例 2: ...
示例 3: ...
示例 4: ...
```

**10+ 个示例 - 可能过多:**
- 占用太多 token
- 可以考虑微调模型

### 示例多样性很重要

覆盖不同的边界情况：

```xml
<示例>
<!-- 正常情况 -->
邮箱: user@example.com → 有效

<!-- 缺少 @ -->
邮箱: userexample.com → 无效

<!-- 缺少域名 -->
邮箱: user@ → 无效

<!-- 特殊字符 -->
邮箱: user+tag@example.co.uk → 有效

<!-- 空值 -->
邮箱: "" → 无效
</示例>
```

## 结构化示例格式

### 使用 XML 标签组织

```xml
<examples>
  <example>
    <input>请求退款</input>
    <category>售后服务</category>
    <priority>高</priority>
  </example>

  <example>
    <input>产品咨询</input>
    <category>售前咨询</category>
    <priority>中</priority>
  </example>

  <example>
    <input>投诉建议</input>
    <category>客户反馈</category>
    <priority>高</priority>
  </example>
</examples>

请分类以下请求: "我想了解价格"
```

### 使用表格格式

```markdown
| 输入文本 | 分类 | 置信度 |
|---------|------|--------|
| "感谢你们的帮助" | 感谢 | 0.95 |
| "这太糟糕了" | 投诉 | 0.88 |
| "如何使用这个功能" | 咨询 | 0.92 |

请分类: "产品质量很好"
```

## 示例 + 说明组合

最佳实践是结合示例和说明：

```xml
<任务>
将客户评论转换为结构化数据

<说明>
提取以下字段:
- 评分: 1-5 星
- 情感: 正面/负面/中性
- 关键词: 最多 3 个
</说明>

<示例>
评论: "产品质量很好，物流也快，五星好评！"
输出: {
  "评分": 5,
  "情感": "正面",
  "关键词": ["质量", "物流", "好评"]
}

评论: "价格贵，功能一般，不推荐。"
输出: {
  "评分": 2,
  "情感": "负面",
  "关键词": ["价格", "功能", "不推荐"]
}
</示例>

请处理: "外观漂亮，但续航不行。"
</任务>
```

## 常见错误

### ❌ 错误 1: 示例不一致

```
示例 1: {"name": "张三", "age": 25}
示例 2: {姓名: "李四", 年龄: 30}  // 字段名不一致
```

### ❌ 错误 2: 示例过于相似

```
示例 1: "很好" → 正面
示例 2: "真好" → 正面
示例 3: "不错" → 正面
// 缺少负面和中性示例
```

### ❌ 错误 3: 示例包含错误

```
示例: 2 + 2 = 5  // 错误示例会误导 Claude
```

## 实践模板

### 分类任务模板

```xml
<分类任务>
根据以下示例，对文本进行分类：

<训练示例>
文本: "[示例1文本]"
类别: [类别1]

文本: "[示例2文本]"
类别: [类别2]

文本: "[示例3文本]"
类别: [类别3]
</训练示例>

待分类文本: "[新文本]"
</分类任务>
```

### 生成任务模板

```xml
<生成任务>
参考以下示例风格，创建类似内容：

<风格示例>
[示例1 - 展示期望的语气和格式]

[示例2 - 展示不同场景]

[示例3 - 展示变化模式]
</风格示例>

创建主题: [新主题]
</生成任务>
```

## 总结

- ✅ 提供 3-5 个多样化的示例
- ✅ 保持示例格式一致
- ✅ 覆盖典型和边界情况
- ✅ 结合示例和说明
- ✅ 使用结构化格式组织
- ❌ 避免过多或过少示例
- ❌ 避免示例质量问题
- ❌ 避免示例之间冲突
