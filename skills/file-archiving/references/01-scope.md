# 定义系统范围 (10-19)

## 概述
该章节指导用户在创建 Johnny.Decimal 系统前明确定义系统边界，确保既不遗漏应包含的内容，也不包含应排除的内容，为后续的区域和类别设计奠定基础。

## 关键步骤
1. **理解系统边界的重要性**:系统的结构完全取决于要组织的内容
2. **评估系统规模**:确定系统的大小是否合适(既不能太小也不能太大)
3. **撰写范围声明**:用 1-5 天时间明确定义"在范围内"(in scope)和"超出范围"(out of scope)的内容
4. **记录目标**:可选地记录为什么做这件事、希望达成什么、完成后的期望感受
5. **定期审查**:将范围声明打印出来,作为后续工作的参考基准

## 核心概念

- **系统范围(System Scope)**: 用几句话定义系统包含什么、不包含什么的边界声明
- **系统规模的类比**:
  - **单个 ID (13.15)**: 具体任务,如"粉刷厨房" - 几个文件,一个周末完成
  - **类别 (43)**: 相关任务集合,如"改造厨房" - 100 个 ID,约一个月完成
  - **区域 (50-59)**: 大型项目,如"重建房屋" - 10 个类别,约 12 个月完成
  - **完整系统**: 超大型项目,如"建设农场" - 多个区域,可运行数年
- **创意模式例外**:对于需要管理大量相似小项目的场景(如广告代理、音效制作),需要为每个项目创建模板,而不是使用标准的区域/类别结构

## 实践要点

- **思考要宽泛**:避免将系统范围定义得过于狭窄,否则容易耗尽可用区域
- **最小化系统数量**:每个额外的系统都会增加心理负担和管理开销,应尽量扩大单个系统的范围
- **暂时不考虑区域和类别**:在完成范围定义之前,不要急于思考具体的区域和类别划分
- **使用实体工具**:
  - 散步时用手机语音备忘录记录灵感
  - 打印工作表并用铅笔填写(推荐 Mitsubishi 9850 铅笔)
  - 将范围声明打印、塑封并贴在墙上作为持续参考
- **区分项目模式**:
  - 标准模式:单个大型项目或日常生活管理
  - 创意模式:大量相似的小型项目(需已有"任务代码"体系)

## 常见问题/注意事项

- **系统可以有多小?**:没有下限,但如果内容太少(只有几项),可能不需要 Johnny.Decimal;考虑它是否只是更大系统的一个"区域"
- **系统可以有多大?**:有上限 - Johnny.Decimal 编号有限,系统过大会缺乏清晰度且难以管理;即使是价值 $60M、10 人团队、3 年周期的大型政府数据中心迁移项目也未用完所有区域
- **在范围内 vs 超出范围**:
  - 在范围内:通常比较明显,用描述性句子列出主要工作内容
  - 超出范围:重点记录那些"可能被误认为在范围内但实际排除"的内容,用于后续澄清和沟通
- **范围蔓延风险**:未明确定义范围会导致遗漏重要内容或错误包含不该管理的内容
- **工作场景特殊性**:如果在工作中使用且不是项目经理,应与项目经理密切合作,使用项目的官方范围声明
