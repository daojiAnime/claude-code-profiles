# 创建区域和类别 (30-39)

## 概述

这个章节指导用户将收集的便利贴（代表需要组织的事物）系统化地组织成 Johnny.Decimal 的两层结构：区域（Areas）和类别（Categories）。核心目标是创建一个陌生人也能轻松使用的、清晰无歧义的分类系统。

## 关键步骤

1. **理解结构约束**：最多 10 个区域，每个区域最多 10 个类别（实际应更少）
2. **选择组织方式**：可以先按类别分组再归入区域，或先按区域分组再细分类别（建议两种方式都尝试）
3. **将便利贴分组**：根据直觉将所有便利贴进行归类
4. **命名区域和类别**：使用清晰、描述性的名称（字数不限，避免歧义）
5. **验证设计**：通过"陌生人测试"检验——一个陌生人能否快速找到特定内容？

## 核心概念

### **区域 (Areas)**

- **定义**：生活或工作的宽泛领域，如"生活管理"、"家庭事务"、"网球俱乐部"
- **数量限制**：最多 10 个，建议使用更少（3-6 个常见）
- **设计原则**：
  - 应足够宽泛，代表"生活的领域"或"专业领域"
  - 避免过度细分，为未来扩展留出空间
  - 对任何人都应该无歧义（例如寻找健康保险单时，选择"生活管理"而非"家庭事务"应该是显而易见的）
- **团队协作**：不同团队成员可以深度理解不同的区域（如项目经理熟悉"项目管理"区域，技术主管熟悉"技术"区域）

### **类别 (Categories)**

- **定义**：相似事物的集合，代表某种"工作类型"，如旅行文件、营销材料、设施维护
- **数量限制**：每个区域最多 10 个类别（典型系统中每个区域 3-6 个）
- **设计原则**：
  - 类别是在区域上下文中的进一步细分
  - 便利贴上的单个条目将成为类别下的具体 ID
  - 更少的类别意味着更少的决策负担
  - 宽泛优于狭窄（可以随时拆分，但合并更困难）
- **容量规划**：每个类别最终可容纳 100 个条目（11.01-11.99）
  - 非事务性工作：如果接近 100 个条目，说明类别过于宽泛
  - 事务性工作（如发票、事件）：使用其原有的 ID 系统

## 实践要点

### 分组方法选择

- **类别优先**：先将所有内容分组为类别，再将类别归入区域（多数人的自然思维模式）
- **区域优先**：先将所有内容分组为区域，再将区域拆分为类别
- **建议**：两种方式都尝试，可能得到完全不同但都有效的结果

### 归类的灵活性

以家庭保险为例：
- 可以是"家庭 > 保险"（保险作为家庭区域下的类别）
- 也可以是"保险 > 家庭"（家庭作为保险区域下的类别）
- 取决于：个人思维习惯、系统的整体结构、职业背景（保险经纪人 vs 房地产中介）
- **关键**：选择一种并坚持使用

### 命名最佳实践

- **使用完整描述**：字数不限，优先清晰无歧义
- **避免过度简化**：人们倾向于用更短的名字，但应抵制这种冲突
- **团队协作**：最常用某个区域的人应在命名上有最大发言权
- **利用索引**：后续的索引（54.10）可以帮助解决剩余的歧义

### 数量控制

- **典型规模**：
  - 工作项目：3-6 个区域，每个区域 3-6 个类别
  - 总类别数：9-36 个（通常约 25 个）
- **经验法则**：
  - 通常不会使用超过系统总容量的一半
  - 应该熟悉每个类别，不会看到某个类别时想"这是什么?"
  - 25 个类别容易记忆，50 个开始困难，100 个几乎不可能

### 类别中的条目数量

- 如果某个类别只有一张便利贴，需要质疑其有效性
- 验证方法：能否想到下一个会放入的东西？如果能，类别有效；如果不能，考虑合并

### 标准化与复用

- 寻找重复的工作模式（这正是专业性的体现）
- 在多个项目中使用相同的类别编号（如项目经理总是处理利益相关者和风险，可以在每个项目中使用相同的类别编号）
- 第一次可能不明显，但第二、三次会成为超能力

## 常见问题/注意事项

### 组织的终点

- **类别是组织的单位**：区域用于组织类别，类别下的 ID 不需要再组织
- **ID 的自然排序**：ID 从 .01 开始递增，自动按创建日期排序
  - 同时创建的事物聚集在一起
  - 最近创建的有更高的编号
  - 通常需要处理的是最近创建的条目
  - 旧的 ID 聚集在一起以备参考，但日常工作中可以忽略
- **足够即止**：到达类别层级时，已经组织得足够好，无需进一步组织

### 处理冲突与歧义

- 生活不会完美地装入分类桶中，每个人有不同的组织观点
- 不要因为一个细节不完美就放弃整个系统
- "足够好"的类别远胜于完全混乱
- 人类大脑能够记忆和适应，允许一定程度的灵活性

### 技术限制

- **文件路径长度**：Microsoft 系统对文件路径长度有限制
- **应对策略**：
  - 仍然使用完整描述名称设计系统（存储在索引中）
  - 根据需要在文件系统中调整名称
  - 技巧：在 SharePoint 上用下划线替换空格可节省两个字符

### 类别爆炸处理

- 如果发现类别数量过多（如 100 个），质疑是否应该合并
- **分而治之**：大型系统中，不同的人深度理解不同的区域
  - 项目经理熟悉"项目管理"区域
  - 技术主管熟悉"技术"区域
  - 个人生活中可能有很少触及的类别（如童年资料归档）
- **核心原则**：减少心理负担，而非用尽所有可用编号

### 事务性 vs 非事务性工作

- **事务性工作**：重复性工作，有固定模式（如发票、事件、变更请求）
  - 通常已有独特的 ID 系统，直接使用该 ID
- **非事务性工作**：其他所有工作
  - 如果接近 100 个条目，说明类别定义过于宽泛

### 卡片分类法

这是一种成熟的组织方法论，想进一步了解可参考 [Wikipedia: Card Sorting](https://en.wikipedia.org/wiki/Card_sorting)

## 示例：Emoji 分类

### 按颜色（区域）→ 类型（类别）

**区域示例**：
- 红色事物
  - 符号（❤️、❌）
  - 家居用品（📌、☎️）
  - 运动（🏓、🥊）
  - 红发人物（👩‍🦰、👨‍🦰）

- 绿色事物
  - 食物（🥒、🍏、🥝）
  - 动物（🐸、🐢、🐍）
  - 植物（🌴、🌲、☘️）

- 蓝色事物
  - Emoji 表情（😨、😱、😭）
  - 符号（♿、🆘、🅰️）
  - 动物（🐋、🐳、🐬）
  - 服装（👕、👔、💄）

### 注意重复类别名称

- 不同区域中可能出现同名类别（如"动物"、"符号"）
- 无需担心：在区域上下文中，歧义已被消除
- 例如：找牛仔裤 → 蓝色区域 → 服装类别（而非表情、符号、动物）
