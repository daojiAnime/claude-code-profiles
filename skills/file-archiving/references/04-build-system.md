# 构建系统 (40-49)

## 概述
这一章节指导用户将前期的概念设计转化为实际可用的 Johnny.Decimal 系统，包括分配编号、创建文件夹结构、迁移现有文件，并建立标准化的组织习惯。

## 关键步骤

1. **复习 Johnny.Decimal 编号规则** (42.01)
   - 系统共有 100 个分类（00-99）
   - 每 10 个分类组成一个区域（如 00-09、10-19）
   - 使用前导零确保排序一致性

2. **为区域和分类分配编号** (43.01)
   - 方法一：直接在文件系统中创建文件夹
   - 方法二：使用文本文件创建结构（便于修改）
   - 仅创建区域和分类层级，暂停后续操作

3. **审视和思考** (43.02)
   - 至少给自己几天时间审视创建的结构
   - 确保分类逻辑符合实际使用场景
   - 虽然后期可以修改，但应尽量一次性设计好

4. **创建 ID** (43.03)
   - 为每个便签纸（代表一个具体事项）创建对应的 ID
   - 合并重复或相关的便签内容
   - ID 格式：`分类编号.序号`（如 12.01）

5. **迁移现有文件** (43.04)
   - 将现有文件移动到相应的 ID 文件夹中
   - 同时整理文件命名规范

## 核心概念

- **分类是核心单元**: 分类（category）是大脑最容易记住的组织单位，系统设计的核心就是让分类易于记忆和查找

- **两层深度原则**: 所有内容必须存储在"ID"层级，不能直接存储在"区域"或"分类"文件夹下，确保系统结构可预测

- **ID 是存储容器**: ID 代表一个具体的任务或事项，应该足够细化但不过度分散。一个 ID 通常包含 1-20 个相关文件

- **编号不表示重要性**: 编号 11 和编号 84 没有优劣之分，但可以利用编号顺序反映自然流程顺序（如项目阶段）

- **标准化模板**: 对于重复出现的结构（如项目管理的十大知识领域），应建立标准编号模板，跨项目复用

## 实践要点

- **容器规则**: 区域和分类文件夹只作为容器，绝对禁止直接在其中存储文件
  ```
  ✗ 10-19 Red things/some-file.pdf  （禁止）
  ✗ 11 Symbols/another-file.docx    （禁止）
  ✓ 11.01 Blood types/file.pdf      （正确）
  ```

- **ID 粒度控制**: 一个 ID 应该代表一个明确的事项，典型包含 1-20 个文件。如果超过几十个文件，考虑拆分成多个 ID

- **子文件夹使用**: 虽然不推荐，但可以在 ID 下创建**一层**子文件夹，注意保持整洁和简洁

- **单文件也建文件夹**: 即使只有一个文件，也应该创建对应的 ID 文件夹，保持结构一致性

- **自然顺序优先**: 如果事项有自然的时间或流程顺序，编号时应反映这种顺序

- **标准模板复用**: 识别可复用的区域/分类结构（如项目管理的 10-19 区域），在多个系统中使用相同编号

## 常见问题/注意事项

**Q: 一个 ID 文件夹应该包含多少文件？**
- 没有硬性限制，但通常 1-20 个文件比较合理
- 超过几十个文件时，说明粒度可能过粗，应考虑拆分

**Q: 真的要为一个文件创建整个文件夹吗？**
- 是的！创建文件夹几乎不花时间，但能保持系统一致性和可扩展性

**Q: 可以在 ID 下创建子文件夹吗？**
- 谨慎使用，仅限一层子文件夹
- 不要回到"文件深埋多层"的混乱状态
- 子文件夹命名建议使用"年-月"格式（如 2024-03）

**Q: 编号是否表示优先级或重要性？**
- 不，编号本身不代表重要性
- 但可以利用编号反映自然顺序（如时间流程）
- 后期发现顺序不理想，影响不大，无需过度纠结

**Q: 如何处理跨项目的标准结构？**
- 识别可复用的模板（如项目管理的 10-19 区域）
- 在所有相关系统中使用相同的编号和结构
- 这样可以跨项目预测文件位置，大幅提升效率
